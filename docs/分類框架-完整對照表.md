# AI 法官 內容爭議分類標準 v1（定版）

> 融合來源：本 doc 舊版 + Confluence 子3 §八/§九（2125529187 · /x/YwCxfg）。
> **範圍：只到「問題分類 + 責任歸屬」**——即「這是什麼問題、屬哪一類、是誰的問題、多嚴重」。
> ⚠️ **不含 action**：recommended_action / owner 執行角色 / exec_platform / 判定層→action 對應，留待 action 階段定義。
> 模型：多維並行（faceted）——一筆問題＝一組維度值，非單一深樹。

## 0. 一筆問題 ＝ 分類向量（多維並行）

| 維度（facet）| 取值 / 層級 | 深度 | 角色 |
|---|---|---|---|
| **症狀**（沿用 KKday 進訊）| tag1 › tag2 › tag3 | 3 層 | 入口（客人說什麼）|
| **旅行階段** | PRE / DURING / POST / OTHERS | 1 層 | 時序切片 |
| **歸因域** ★ | 6 域 › 子類(~6) | 2 層 | 問題根因類別 |
| **責任方** ★ | 供應商 / 平台 / 客服 / PM(內容法典) / 客人 | 1 層 | **歸類到誰** |
| **失效型態** | 缺漏 / 寫錯矛盾 / 模糊 / 履約不符 / 服務未達 / 誤解 | 1 層 | 問題形態 |
| **內容面向**（僅 ①）| 8 面向 | 1 層 | 內容問題定位 |
| **欄位 / Rule**（僅 ① 深鑽）| 60 欄 › R1-1~R5-5 | 2 層 | 最細粒度 |
| **verdict 問題性質** | 六分類 | 1 層 | 問題類型標籤 |
| **嚴重度** | P0–P3 | 1 層 | 問題輕重 |
| **信心度** | 0–1 | 連續 | 判定可信度 |

**分類向量範例（到此為止，不往 action 走）**
```
{ 症狀: 商品體驗>現場工作人員>司機導遊,  階段: DURING,
  歸因域: ②供應商履約>語言服務,  責任方: 供應商,
  失效型態: 履約不符,  面向: —(非內容),  Rule: R1-6(導遊語系),
  verdict: contract_breach,  嚴重度: P1,  信心: 0.85 }
```

## 1. 歸因樹（6 域 → 子類 → 具體問題）— 核心

```
客訴根因歸屬（6 大歸因域 · L1）
├─ ① 商品內容問題　[責任方: 供應商(內容)+PM(法典)｜verdict: content_missing/unclear/real_config]
│   ├─ 商品定位  ├─ 行程流程  ├─ 費用資訊  ├─ 集合資訊
│   └─ 使用兌換  ├─ 成團條件  ├─ 限制與風險  └─ 承諾與SLA
│       └─ 深鑽：8面向 → 60欄位 → Rule ID（R1-1~R5-5）★唯一深鑽支
├─ ② 供應商履約問題　[責任方: 供應商(現場)｜verdict: contract_breach｜需訂單佐證]
│   ├─ 現場人員  ├─ 集合執行  ├─ 語言服務
│   └─ 成團履約  ├─ 設備車餐  └─ 安全SOP
├─ ③ 訂單與交易問題　[責任方: 平台/客服｜verdict: escalate_ops]
│   └─ 訂單確認 / 修改 / 取消 / 退款 / 付款 / 憑證單據
├─ ④ 平台與系統問題　[責任方: 平台(RD/PM)｜verdict: escalate_ops]
│   └─ 頁面App bug / 搜尋篩選 / 流程設計 / 通知失效 / 行銷優惠
├─ ⑤ 客服與營運問題　[責任方: 客服/營運｜verdict: escalate_ops]
│   └─ 回應SLA / 處理態度 / 補救SOP / 跨單位協調
└─ ⑥ 客人理解/期待落差　[責任方: 客人(非問題)｜verdict: customer_misread]
    └─ 內容清楚誤解 / 期待過高 / 看錯方案 / 未讀注意 / 不可抗力
```

## 2. 責任歸屬總表（歸類到誰）

| 歸因域 | 責任方 | verdict（問題性質）| 典型嚴重度 |
|---|---|---|---|
| ① 商品內容 | **供應商（內容）+ PM（法典）** | content_missing / content_unclear / real_config_issue | P0–P3 |
| ② 供應商履約 | **供應商（現場）** | contract_breach | P0–P2 |

## 3. 各域 子類 × 具體問題 × 失效型態（分類明細）

### ① 商品內容問題（責任方：供應商內容 + PM 法典）
| 子類＝8面向 | 具體問題（失效型態 × 欄位）| Rule | verdict | 嚴重度 |
|---|---|---|---|---|
| 商品定位 | 名稱促銷誇大 / 摘要特色不符 / 圖文錯位（寫錯矛盾·模糊）| R5-1~4 | real_config / content_unclear | P2-P3 |
| 行程流程 | 步驟·時長·交通·**導遊語系**·地圖缺漏（缺漏·模糊）| R1-1~6 | content_missing / unclear | P2 |
| 費用資訊 | 包含不清 / **現場強制費未揭露** / 兒童價缺（缺漏）| R4-1 | content_missing / real_config | P1-P2 |
| 集合資訊 | **地點/時間/方式不清** / 識別缺 / 遲到規範缺（模糊·缺漏）| R2-1~9 | content_unclear / missing | P1-P2 |
| 使用兌換 | 使用方式 / 憑證 / 證件 / 語系 缺（缺漏）| R2-10 | content_missing / unclear | P2 |
| 成團條件 | **誤導保證成團** / 最低人數·確認時間缺（寫錯·缺漏）| R3-1~3 | real_config / content_missing | **P0-P1** |
| 限制與風險 | 年齡·健康·孕婦·安全·裝備·保險 缺（缺漏）| R4-1+NOTICE | content_missing | P1-P2 |
| 承諾與SLA | **FFC前承諾** / **退款SLA缺** / 異動通知缺（寫錯·缺漏）| R3-6~10 | real_config / content_missing | **P0-P1** |

### ② 供應商履約問題（責任方：供應商現場；失效型態＝履約不符；需訂單佐證）
| 子類 | 具體問題 | verdict | 嚴重度 |
|---|---|---|---|
| 現場人員（司機/導遊）| 態度差 / 未到 / 不專業（症狀最大宗 58%）| contract_breach | P1-P2 |
| 集合執行 | 司機未到/遲到 / 地點與內容不符 | contract_breach | **P0-P1** |
| 語言服務 | 導遊語言與內容承諾不符 | contract_breach | P1 |
| 成團履約 | 已承諾成團卻臨時取消/未達 | contract_breach | **P0** |
| 設備車餐 | 規格與內容承諾不符 | contract_breach | P2 |
| 安全SOP | 安全措施未落實 | contract_breach | **P0** |

### ③ 訂單與交易問題（責任方：平台/客服；失效型態＝服務未達/系統錯誤）
子類：訂單確認 / 訂單修改(異動·客製化·旅規) / 訂單取消 / 退款 / 付款交易 / 憑證單據 → verdict escalate_ops。*退款/SLA「內容」缺 → 回落 ①*

### ④ 平台與系統問題（責任方：平台 RD/PM）
子類：頁面App bug / 搜尋篩選 / 流程設計 / 通知失效 / 行銷優惠 → verdict escalate_ops。*活動文案不清 → 回落 ①*

### ⑤ 客服與營運問題（責任方：客服/營運）
子類：回應SLA / 處理態度 / 補救SOP / 跨單位協調 → verdict escalate_ops。

### ⑥ 客人理解/期待落差（責任方：客人，非問題）
子類：內容清楚但誤解 / 期待過高 / 看錯方案 / 未讀注意事項 / 不可抗力 → verdict customer_misread。

## 4. 症狀 → 歸因 裂解（同症狀可裂多域）

| KKday 症狀 tag | 裂解到（責任方）| 判準 |
|---|---|---|
| 現場工作人員 › 司機/導遊（58%）| ②供應商 / ①供應商內容·PM / ⑥客人 | 寫了沒做到=② ／ 沒寫=① ／ 寫清楚沒看=⑥ |
| 商品資訊確認 | ①內容 / ⑥客人 | 缺漏模糊=① ／ 誤解=⑥ |
| 訂單確認問題（51%）| ③平台·客服 / ①(少) | 系統客服=③ ／ SLA內容寫錯=① |
| 訂單申請修改（異動/客製化/旅規）| ③平台 / ⑤客服 | 流程=③ ／ 處理不力=⑤ |
| 客人會遲到/去錯集合地 | ①(集合不清) / ⑥(沒看) | 集合資訊模糊=① ／ 寫清楚沒看=⑥ |
| 付款與交易 | ④平台 / ③ | 金流bug=④ ／ 流程=③ |
| 行銷優惠相關 | ④平台 / ①(文案) | 優惠失效=④ ／ 文案不清=① |

## 5. 嚴重度 Rubric（P0–P3，問題輕重）
- **P0-P1 重**：影響成行確定性 / 資金安全 / 誤導性承諾（誤導保證成團、退款SLA缺、FFC前承諾、成團履約失敗、安全SOP）。
- **P2 中**：關鍵資訊缺漏/模糊但未直接誤導（最低人數模糊、確認時間無時區、集合/費用不清）。
- **P3 輕**：格式/可讀性瑕疵。

## 6. 細化原則 + schema 落地欄位（分類用）
- **細化只在 ① 商品內容深鑽**：`面向(8) → 欄位(60) → Rule ID`；②~⑥ 兩層（域→子類）即可。
- 要再細化＝加新維或維內加值，不往單樹底下生層。
- **schema（分類欄位，TicketFinding）**：`symptom_tag1/2/3 · trip_stage · root_cause_domain · responsible_party · sub_cause · failure_type · dimension · suspected_field · hit_rule_id · verdict · severity · confidence`。
- ⚠️ **action 階段欄位（暫不納入）**：`judgment_tier · recommended_action · owner_role · exec_platform · writer_handoff`。
