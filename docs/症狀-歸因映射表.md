# 症狀 → 歸因 候選集映射表（§十 草案）

> 接起兩棵樹：**軸A 進線症狀樹**（KKday 既有 tag1›tag2›tag3，判決前可給）→ **軸B 歸因判決樹**（6 域，判決後收斂）。
> 用法：進線時依症狀查本表得「候選歸因域集合 {1..N}」（描述性、可在判決前給）；判決時依「收斂判準」讀商品頁+訂單，從候選集挑單一歸因域 + 責任方 + verdict。

## MECE 反向驗證（6 域 vs KKday 症狀全集）

✅ **無孤兒**：29 個 tag3 全部能映射進 6 域，6 域對 KKday 官方症狀全集 MECE 覆蓋成立。

⚠️ **三個結構性發現（直接支撐階段化架構）**：

1. **⑤客服營運 在症狀樹裡沒有獨立分類**。客服表現散在各 tag2 的「處理不力」子情境（如退款拖延、異動未處理）。症狀樹是「客人問什麼」視角，沒有「客服表現」軸 → **⑤ 永遠只能在判決階段浮現**，進線症狀打不出來。**證明 ⑤ 必須判決後才給。**

2. **進線 83% 是訂單類（③主導），是數據源偏差不是真實分布**。chatbot + order_message 兩個源天生繞著訂單轉。**①商品內容的問題大量「沉默」在「商品資訊確認」(14.38%) 與「現場工作人員」(7.17%) 裡，且必須讀商品頁判決才浮現** → 別被進線症狀的 ③ 大佔比誤導成「商品內容沒問題」。

3. **大量症狀是「確認/詢問」型，本質分不清 ①缺漏 vs ⑥誤解**。最大宗「訂單/憑證內容確認」(39k)、「確認聯繫方式」(24k)、「接送時間地點確認」(22k) 都是客人「來問」——可能內容沒寫清楚(①)、也可能寫了想再確認(⑥)。**這是全 dataset 最大的「需證據才能判責任方」證據** → 直接驗證進線只能給候選集、判決才收斂。

## 落地：schema 對應
- 進線寫入：`symptom_tag1/2/3` + `root_cause_candidates[]`（查本表得 {1..N}）
- 判決寫入：`root_cause_domain`（從候選收斂單選）+ `responsible_party` + `verdict` + `failure_type` + `confidence`
- 收斂所需證據：商品頁(8面向/60欄) + 訂單(時間地點/狀態) + 供應商回覆 → `evidence_level`
