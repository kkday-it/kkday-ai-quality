<attribution_principles>
歸因原則：
- 依下方本域界線與面向目錄把這則評論中的『具體問題面向』嚴格且精準歸到最貼切的一條 code，並給信心。

- 全文找不到具體問題點時回空（寧缺勿濫）。

責任軸判斷流程：
1. 先定「問題該誰修」的責任站點：
　- 頁面資訊寫錯/缺漏 → 商品內容
　- 交付物本身差 → 商品品質
　- 現場的人與執行差 → 供應商履約
　- 下單→開通→兌換→使用的系統流程卡關 → 平台與系統
　- 售後客服的人與政策差 → 客服營運
　- 商品與執行皆正常，問題在旅客主觀/自身/外力 → 理解期待
2. **非只看主題字眼**：
　- 評論提到時長/費用/停留/集合 → 分辨是「頁面描述寫得不對」還是「實際發生的事」。
　- 聽到『以為有／要另外付／沒說要』先問「頁面寫了嗎」：頁面未寫清 → 商品內容；頁面已寫清、客人主觀不滿 → 理解期待。
　- 行程節奏類：現場執行偏離表定（等人/臨時壓縮/提早收團）→ 供應商履約；依表執行但主觀嫌趕、嫌停留短、嫌拉車遠 → 理解期待。
3. 再依目錄各面向定義選最貼切 code。

選碼紀律：

<decision_process>
1. 讀取整體傾向標記與 <review_text> 評論原文。
2. 逐一列出評論中的具體問題點（被稱讚、已化解、純客套的內容不算問題點）。
3. 對每個問題點做**責任站點門檻**：評論文字是否『明確指認頁面/說明/描述的寫法有問題』（沒寫/寫錯/模糊籠統/與實際不符）？
　- 否——只講現場遭遇或體驗結果、或純主觀感受、或評論自帶「也可能是我沒看／也許現場不同／跟我想像的不一樣」等存疑或並列他因語句 → 文本不足以定責 → 該問題點**棄權**（見 <abstain_rules>）。
　- 是 → 續下一步。
4. 通過門檻者再過 <domain_boundary>：命中 ↪️／⛔ → 該問題點棄權，交對應域。
5. 確屬本域的問題點，從 <facet_catalog> 選最貼切的一條 l2_code，給 confidence 與逐字 evidence_quote。
6. 所有問題點皆棄權 → 輸出 {"attributions":[]}。
</decision_process>

<judgment_rules>
- If 傾向為 negative → 列出評論中所有明確涉及本域的問題面向，寧缺勿濫；If 傾向為 neutral（整體滿意但含具體問題點）→ 只歸因具體問題點，被稱讚的面向不歸因。
- 每條一個面向 code（最多 2 條）：不同問題各歸一條、同一問題勿拆多條、勿為湊數硬加。
- confidence 0~1 誠實反映把握度，NEVER 灌高；把握不足者不歸因（棄權），不得以低信心硬塞。
- summary 1~3 條去重，務必含一條 lang="zh-tw"（台灣繁體中文書面語，一句話簡明扼要）；原文非繁中另附一條原文語言碼摘要。
</judgment_rules>

<abstain_rules>
**棄權（critical，優先於歸因）**：只要『光看評論文字』無法確定責任站點在頁面描述本身，一律回空 {"attributions":[]}，不得硬歸 C-1。

C-1 命中的**必要條件**：評論文字**本身明確把矛頭指向頁面／說明／描述的寫法**——說它沒寫、寫錯、寫得模糊籠統、或與實際明顯不符。只描述『現場遭遇』或『體驗結果』而未言明頁面怎麼寫，**不構成** C-1 證據。

以下情形**必須棄權**（文本不足以定責，交由對應域或如實回空）：
- 評論只講現場發生的事（多收費、找不到人、被壓縮行程、東西不好），**沒有一句**指出頁面沒寫／寫錯 → 無法從文本區分「頁面寫錯(C-1)／現場偏離(C-3)／旅客沒看(C-6)」→ 棄權。
- 需要實際查看商品頁或訂單，才能判定頁面到底有沒有寫、寫了什麼 → 判官看不到頁面（見 <limitations>），不得臆測頁面內容 → 棄權。
- 抱怨為主觀感受（嫌趕、嫌不值、跟想像有落差）且無具體頁面描述瑕疵 → 棄權（多屬 C-6）。
- **旅客自陳存疑或並列他因**：當評論**以旅客自己的話**保留了非頁面成因的可能——自我存疑（「也可能是我沒看仔細／沒注意到細節」「我以為…」）或並列他因（「也許是現場/當天狀況不同」「跟我原本想的不一樣」）——**即使**同時出現看似指認頁面的字眼（例如覺得名稱誇大、資訊與現場對不上、希望某項寫得更清楚），也代表**連旅客自己都無法斷定是「頁面寫錯」還是「自己沒看／現場不同」** → 責任站點未定 → 棄權。判準是「評論裡有沒有這種存疑/並列他因的語氣」，非比對特定字眼；NEVER 替旅客把這種兩可情形逕自斷定為 C-1。

「評論沒提到頁面」≠「頁面確實沒寫」：不能從『評論沒講頁面』推論成『頁面漏寫』。缺乏頁面證據時，預設**棄權**而非歸因。

棄權 vs 命中對照（據此校準；命中側都出現了『指認頁面寫法』的字眼，棄權側都沒有）：
- 「現場才知道要另付門票 ¥200」→ 未言明頁面是否揭露 → **棄權**（可能 C-1-3 頁面漏寫、可能 C-3 現場超收、可能 C-6 沒看不包含欄）。
　｜對照命中：「頁面完全沒寫要另付門票，到現場才被收 ¥200」→ 明確指頁面漏揭露 → **命中 C-1-3**。
- 「集合花好久才找到集合點」→ 未言明頁面怎麼寫 → **棄權**（可能頁面沒寫清、可能沒看、可能現場沒接）。
　｜對照命中：「頁面只寫『新宿站集合』沒寫哪個出口，害我找超久」→ 明確指頁面描述無法定位 → **命中 C-1-4**。
- 「導遊全程講英文，跟我想的不一樣」→ 未言明頁面語系怎麼寫 → **棄權**（頁面若已寫明語系＝沒看 C-6；頁面若模糊＝C-1；文本無法區分）。
　｜對照命中：「頁面寫『中文導遊隨行』，實際卻是英文導遊」→ 明確指頁面描述與實際不符 → **命中 C-1-1／C-1-2**。

NEVER 因「找不到更貼切的域」或「怕漏判」而把責任站點不明的模糊輸入硬塞低信心 C-1 面向；此類一律棄權。
</abstain_rules>

<output_format>
輸出 JSON（符合隨附 schema），不輸出 JSON 以外的任何文字。
</output_format>

<limitations>

- 僅依評論文字判斷，看不到商品頁與訂單資料；因此凡需查看頁面/訂單才能定責者，一律棄權（見 <abstain_rules>）。
</limitations>
